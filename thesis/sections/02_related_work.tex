\chapter{Related Work}

This chapter reviews only the work that motivates the study design. It is
organized around five themes: algorithm auditing, LLM behavior and access
surfaces, safety and refusal calibration, control and factuality tasks, and the
reliability of LLM measurement. It then positions the thesis. Each section states
what it contributes to an access surface audit and what it leaves open. The
empirical prompts group into six behavioral families (Categories A through F in
Chapter 3), described here by behavioral function rather than category letter.

\section{Algorithm Auditing}

Algorithm auditing supplies both the method this thesis adopts and its guiding
principle. It probes an opaque deployed system through controlled inputs and
observable outputs, and it makes the deployed product, not an abstract model, the
unit of analysis. Auditing has been framed as a way to detect differential
treatment on platforms whose internal mechanisms are unavailable
\parencite{sandvig_auditing}. The logic transfers to proprietary LLMs. An
evaluator cannot see hidden instructions, tool policies, routing, or moderation,
but can still compare outputs collected under documented conditions. Product
level audits show why the
deployed system is the right unit. The \emph{Gender Shades} audit
\parencite{gender_shades} exposed intersectional accuracy gaps in commercial
classifiers, gaps that aggregate accuracy scores had concealed. Publishing such audits can change
vendor behavior \parencite{raji_actionable}. A deployed healthcare tool was also
shown to contain racial bias \parencite{obermeyer_bias}. Accountability work
makes the system boundary explicit by documenting evidence, scope, and deployment
assumptions together \parencite{raji_gap}. Broader sociotechnical critiques warn
that evaluations fail when they abstract away deployment context
\parencite{selbst_abstraction,suresh_guttag}. A consumer app is therefore part of the deployed
condition, not a transparent wrapper around an endpoint. What this literature does
not supply is a method for auditing \emph{generative, nondeterministic} text
systems, where the same input can yield different outputs. That problem is taken
up in Section~\ref{sec:llm-measurement-reliability}.

\section{LLM Behavior, Deployment, and Access Surfaces}

This section establishes the premise the thesis rests on. Assistant behavior is a
property of the deployed system, and the access surface is itself an empirical
variable. Modern LLM behavior is shaped by far more than pretraining. GPT-3
established prompt based task performance \parencite{gpt3}. InstructGPT showed
that instruction tuning with human feedback reshapes what a model will and will
not do \parencite{instructgpt}. Anthropic's helpful, honest, and harmless work
and Constitutional AI make the same point for safety
\parencite{helpful_harmless,constitutional_ai}. Assistant behavior
is a trained, system level property, not raw next token continuation. Foundation
model risk work adds that risks depend on how models are adapted and deployed
\parencite{foundation_models,weidinger_risks}. The deployed system, not the bare
model, is what an audit must measure. Benchmarks are the standard
instrument. MMLU \parencite{mmlu}, GSM8K \parencite{gsm8k}, HumanEval
\parencite{codex}, BIG-bench \parencite{bigbench}, and HELM \parencite{helm} are
prominent examples. They are almost always run through programmatic APIs, which
leaves open whether the
behavior they measure is the behavior a user receives through an app. A small but
growing literature treats the access surface itself as the variable. The closest
precedent is \emph{Dual Standards} \parencite{dualstandards}, which gives
empirical evidence that moderation differs between API and WebUI access. It
finds the WebUI \emph{more} restrictive, the opposite of the direction found here,
which is exactly why the claim here is only that the surfaces differ, not that they
differ in a fixed direction. Two further results make the app a moving, opaque
target. The same named service can change within months, and consumer systems can
wrap the model in partly extractable hidden prompts
\parencite{chatgpt_drift,prompt_extraction}. What remains missing, and
what this thesis supplies, is a direct, paired comparison of a bare API against
the official app on one fixed diagnostic prompt bank.

\section{Safety and Refusal Calibration}

This section grounds the high risk, boundary, and over-refusal prompts in the
bank and the coding distinctions a refusal audit needs. Safety benchmarks supply the
high risk and boundary prompts. Do-Not-Answer \parencite{donotanswer} curates
prompts a responsible model should decline. HarmBench \parencite{harmbench}
standardizes refusal evaluation. AILuminate \parencite{ailuminate} organizes
product-level risk categories. Moderation classifiers such as Llama Guard
\parencite{llama_guard} show that a provider can place a filter between user and
model that, from a black-box view, is part of the deployment stack. Refusal
benchmarks sharpen refusal and partial answer coding. SORRY-Bench \parencite{sorrybench},
StrongREJECT \parencite{strongreject}, and WildGuard \parencite{wildguard}
evaluate refusal and jailbreak behavior, while RefusalBench
\parencite{refusalbench_bio}, in the biological-research domain, supplies the
key caution: a single refusal rate can misrank systems when safe, useful
engagement is possible. This thesis therefore
distinguishes full refusal, partial refusal, and safe redirection rather than
treating every non answer alike. Over-refusal work motivates the benign half of
the design. XSTest \parencite{xstest} targets exaggerated safety on clearly safe
prompts with risky surface words, and OR-Bench \parencite{orbench} scales over-refusal evaluation. Red teaming and jailbreak research is background rather than
a prompt source \parencite{ganguli_redteam,redteam_lms,universal_attacks,pair}.
This thesis uses
plain diagnostic prompts so that any differences read as deployment surface
effects rather than jailbreak artifacts.

\section{Control and Factuality Tasks}

Beyond safety, the bank needs control families: instruction following,
factuality, and socially-sensitive inference. These separate deployment surface
effects from ordinary task variation. IFEval
\parencite{ifeval} motivates instruction following items with explicit, checkable
constraints that can be scored without subjective judgment, and it flags a
practical app evaluation hazard: rendered markdown or transcription artifacts can
mimic format failures unless screenshots are checked, which is why format compliance
divergences are coded conservatively here. Factuality fills three roles.
Short answer benchmarks SimpleQA and TruthfulQA define when a model should
simply be correct \parencite{simpleqa,truthfulqa}.
Hallucination studies
\parencite{maynez_factuality,hallucination_survey,factscore,selfcheckgpt}
motivate separating unsupported memory answers from evidence supported ones.
Retrieval work shows that adding visible sources changes the factual support a
user receives \parencite{dpr,rag,webgpt,verified_quotes,blenderbot2}. Because the app can
display sources while the bare API cannot, visible sources are coded as a
deployment surface signature rather than dismissed as cosmetic. Socially-sensitive
inference is a negative control. Work on bias in embeddings and generated text
\parencite{caliskan_bias,debiasing_embeddings,sheng_bias,realtoxicityprompts}
shows that models absorb demographic associations. Bias claims require explicit
normative grounding \parencite{blodgett_survey}, and BBQ \parencite{bbq} is
built so the correct response is often to decline an
unsupported demographic inference. This thesis therefore codes ``cannot
determine from the information given'' as correct rather than as a refusal, so
these controls do not inflate the refusal result.

\section{Reliability of LLM Measurement}
\label{sec:llm-measurement-reliability}

Auditing a generative system raises two measurement problems a classification
audit does not: nondeterminism and coder subjectivity. This section
motivates the repetition and reliability machinery in Chapter 3. First, outputs
are nondeterministic. The same prompt can produce different responses across runs
and hardware from floating point and batching effects alone, before any deliberate
sampling \parencite{llm_nondeterminism}, so an
audit that collects a single response per condition can mistake run-to-run noise
for an access surface difference. This is why this thesis repeats the API
condition and reports run-to-run stability. Second, scaling the coding of
open ended outputs invites automated judging, but MT-Bench \parencite{mtbench}
and G-Eval \parencite{geval} document sensitivity to prompt wording and answer
position, so automated labels need human checking rather than blind trust. Third,
human coding must be quantified. Kappa coefficients are the standard instruments,
with established interpretation guidance
\parencite{cohen_kappa,fleiss_kappa,mchugh_kappa}. This
thesis combines all three: repetition for nondeterminism, automated coding
checked against an independent blind second coder, and kappa based inter rater
reliability. A reported divergence is therefore unlikely to be an artifact of
stochasticity or coder subjectivity.

\section{Positioning of This Thesis}

Taken together, the cited work establishes three points and leaves one open. LLM
behavior is a property of the deployed system rather than the bare model. The
access surface can change that behavior, and measuring a generative system
requires controlling for nondeterminism and coder subjectivity. What it does not
provide is a compact, evidence backed, paired audit that holds a fixed diagnostic
prompt bank constant while directly comparing a bare OpenAI API condition with the
official ChatGPT app, coding only observable differences with screenshot-backed
evidence and reliability controls. That is the contribution of this thesis.

The same framing bounds the claims. The study can show that observable outputs
differ under documented access conditions, but it cannot identify the hidden
system prompt, retrieval policy, tool routing decision, or moderation component
behind a given app output. Naming those components is left to future work. The
implication that survives is narrow and practical. LLM evaluations should name the
access surface they measure and validate app behavior directly rather than
assuming it from API results.
