Auditing Access Channel Effects in LLM Behavior
Motivation
ManyevaluationsoflargelanguagemodelsareconductedthroughAPIsbecauseAPIaccessiseasiertoautomateandcontrol.
Ordinary users, however, often encounter the same nominal model through an official consumer application or a third party
wrapper. These channels may add system instructions, visible and invisible guardrails, memory or session handling, retrieval
behavior, interface defaults, rate limits, and wrapper specific orchestration. This is especially important for refusal and safety
behavior: an API result may not reflect what a user experiences in a web interface if additional guardrails or interface level
defaults are applied.
The thesis asks whether conclusions from API based audits transfer to ordinary user access channels: when a study reports that a
modelbehavesinacertainway,isitmeasuringthemodelfamilyitself,ortheaccessenvironmentthroughwhichthatmodelis
reached?
This remains underexplored and fits the Social Computing Group’s broader algorithm auditing methodology: fixed inputs,
documented collection conditions, controlled comparisons, and careful interpretation of platform dependent outputs. The central
research question is: Do nominally identical or closely related LLMs produce systematically different observable behavior
when accessedthrough APIs, consumerapplications,andwrapper interfaces?

Proposed Protocol
Thethesiswillbeacompactaccesschannelaudit,notanopenendedbenchmark. Thecoredesignisdeliberatelyconservative:
oneprimarymodelfamilywillbeevaluatedthroughthedirectAPIandthecorrespondingconsumerapplication. Athirdparty
wrapperwillbeincludedonlyiftheunderlyingmodelisidentifiableanddatacollectionistechnicallyandethicallyfeasible. If
time permits, one or two additional model families may be used as robustness checks, but they are not required for the main
contribution.
Thepromptsuitewillbedevelopedintwostages. First,apilotsetof15to20promptswillbeusedtocheckfeasibility,logging
quality, and coding categories. After the pilot, the final prompt suite will be frozen at approximately 60 to 80 prompts. The suite
willcoverrefusalandguardrailingbehavior,safetyframing,sociallyrelevantorrecommendationprompts,neutralfactualcontrols,
and instruction following taskswith specified output formats.
Repeated measurements will be used where they are most informative. The plan is to run two or three repetitions for refusal
and guardrailing prompts and other categories that show early divergence. If the collection burden becomes too high, lower
risk control prompts will be measured once under controlled conditions. The interpretation will remain cautious: the thesis
willcharacterizeobservabledivergencesandplausibleaccesslayerexplanations,butitwillnotclaimto identifyhidden system
prompts or internal guardrailmechanismsthat are notobservable.
Collection Strategy
APIcollection willuse a reproduciblePython script that recordsprompt ID, modelname and version where visible, timestamp,
controllable settings, raw output, and metadata in structured JSON and CSV files. The consumer application channel is treated as
themainpracticalriskintheprotocol. Browserassistedcollectionwillbetestedunderordinaryaccessconditions,butautomation
is not a methodological dependency. If automation is unstable because of login, rate limits, rendering variability, or platform
restrictions, the thesis will use a controlled manual or assisted manual protocol instead. The same account and access condition
willbeusedthroughout;chatsessionswillberesetwherepossible;promptorderwillbefixedorrandomizedaccordingtothe
final protocol; outputs will be timestamped; screenshots or exported logs will be kept when useful; and no CAPTCHA bypassing
orcircumventionofproviderrestrictionswillbeattempted. Responsedatawillbestoredforanalysisonlyinwayscompatiblewith
provider terms.
Analysis Plan
Theanalysiswillcombinequantitativecomparisonwithqualitativecaseinspection. Quantitativemeasuresmayincluderefusal
and compliance labels, response length, semantic disagreement, hedging markers, safety framing, and output format compliance.
Thresholds for similarity, effect sizes, or agreement between coders will be calibrated after the pilot rather than fixed prematurely.
Qualitatively,thethesiswillinspecthighdivergencecasessuchasAPIcompliancewhiletheapprefuses,strongerwebsafety
framing,wrapper rewriting,andfactual or recommendationdifferences across channels. Thefinalresult should be a divergence
taxonomy withclear limitations.
Candidate Groundwork
Since 1 March 2026, thesis relevant groundwork has been carried out in a forked open source AI application generator codebase.
This workisnot presentedascompletedthesisresults. Itsrelevanceis practical: itprovidespriorexperience withthe kindof
access layer,logging,provider routing, andreliability infrastructure needed for thisaudit.
Access layer implementation. The codebase contains a chat based LLM interface, a backend agent system, prompt to blueprint
generation,modelandproviderselection,WebSocketbasedgenerationstate,livepreviewfeedback,andwrapperlogicaround
multiple model providers. Work in this setting is directly relevant to the thesis because it shows how user facing outputs are
shaped byorchestrationcode, UI defaults, provider selection,session state, andbackendsettings in addition to the modelitself.
Measurement and reliability infrastructure. The March and April work also involved structured logging, retry and rate
limithandling, stale connection safeguards, gateway based provider analytics, model and cost summaries, debugging tools, and
productionreadinessfixesaroundchatandgenerationflows. Theseelementsmapontotheproposedthesispipeline: stableprompt
execution, metadata capture, response logging, model or provider visibility where available, and post hoc inspection of divergent
outputs.
The MSc contribution remains separate and clearly defined: a frozen prompt suite, controlled collection across access channels, a
coded divergence taxonomy, andempirical analysis of observableaccess channeleffects.
